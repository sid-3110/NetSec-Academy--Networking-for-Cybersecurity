01 — INTRODUCTION TO NETWORKING
Understand the fundamental concepts of how devices communicate in a network — the bedrock of all cybersecurity knowledge.
📖 Definition

A network is a collection of two or more computing devices connected together to share resources, data, and services. Think of it like a postal system — every device has an address, messages (data) are packaged, routed, and delivered.

💡 Real World Example

When you stream YouTube, your laptop (device 1) sends a request to Google's server (device 2) over a network. The video travels through multiple routers across the internet — each one reading your device's "address" and forwarding data closer to you. That entire process is networking.

Why Networking Matters in Cybersecurity

Every cyberattack travels over a network. SQL injection, phishing, ransomware, DDoS — they all use networking protocols. If you don't understand how data moves, you cannot understand how it's attacked or defended.

🔴

Offensive Security

Pentesting requires port scanning, packet sniffing, and exploiting network-level vulnerabilities.

🔵

Defensive Security

SOC analysts read network logs, detect anomalies, and trace attack paths through the network.

🕵️

Forensics

Incident responders reconstruct attacks from network traffic captures and firewall logs.

☁️

Cloud Security

Cloud misconfigurations expose network services — understanding ports and IPs is essential.

02 — HOW THE INTERNET WORKS
Trace the full journey of data from your browser to a web server and back — understanding every step.
📖 Definition

The Internet is a global system of interconnected computer networks that use the TCP/IP protocol suite to communicate. It's not owned by any single entity — it's a massive, decentralized collection of networks that all agree to speak the same language.

The Client–Server Model
💡 Analogy — Restaurant

You are the client (customer). The website's server is the server (kitchen). Your browser sends a request (order). The server sends back a response (food). The network is the waiter carrying messages between you.

  CLIENT-SERVER MODEL

  ┌──────────────┐     HTTP Request      ┌──────────────────┐
  │   YOUR       │ ─────────────────────►│   WEB SERVER     │
  │   BROWSER    │                       │  (google.com)    │
  │  (Client)    │◄─────────────────────│  192.178.52.1    │
  └──────────────┘     HTTP Response     └──────────────────┘
                       (HTML, CSS, JS)

  Client responsibilities: Send requests, render responses
  Server responsibilities: Listen for requests, process, respond
  Network:               Route packets between them
What Happens When You Visit a Website

This is the complete step-by-step flow — understanding this is essential for both web security and network analysis.

1

You type "google.com" and press Enter

Your browser needs to find Google's IP address. It can't route to a domain name directly — it needs the numeric IP. It first checks its local cache.

2

DNS Resolution — Domain → IP Address

Browser asks a DNS resolver (usually your ISP's or 8.8.8.8): "What is the IP for google.com?" The resolver returns 142.250.80.46. Now your browser knows where to connect.

3

TCP Connection — Three-Way Handshake

Your browser connects to port 443 (HTTPS) on Google's server. A TCP handshake happens: SYN → SYN-ACK → ACK. Connection established.

4

TLS Handshake — Encryption Setup

For HTTPS, a TLS handshake negotiates encryption. Server sends its certificate. Your browser verifies it's legitimate and sets up an encrypted tunnel.

5

HTTP Request Sent

Browser sends: GET / HTTP/1.1 — requesting the homepage. Headers include your browser type, accepted languages, cookies, etc.

6

Server Processes and Responds

Google's server receives the request, generates the response (HTML, CSS, JavaScript, images), and sends it back with status code 200 OK.

7

Browser Renders the Page

Your browser parses HTML, downloads linked resources (images, scripts), runs JavaScript, and displays the page. The entire process takes milliseconds.

     │
     │ 1. "What's the IP for google.com?"

  [DNS Resolver] → [Root NS] → [.com TLD NS] → [Google NS]
     │                                               │
     │ ← ← ← ← ← ← 142.250.80.46 ← ← ← ← ← ← ← ◄─┘
     │

  Security Note: Each step can be attacked:
  DNS Spoofing    → Redirect to fake IP
  MITM Attack     → Intercept traffic between you and server
  SSL Stripping   → Downgrade HTTPS to HTTP
  BGP Hijacking   → Reroute internet traffic at ISP level
⚠️ Attacker Insight

Every one of these 7 steps is a potential attack point. DNS spoofing redirects you to a fake site. Man-in-the-Middle attacks intercept traffic between steps 3–5. Understanding the normal flow helps you spot the abnormal — which is exactly what security analysts do.

03 — IP ADDRESSING
Every device needs an address. Learn the complete IP addressing system — from IPv4 structure to subnetting.
📖 Definition

An IP address (Internet Protocol address) is a unique numerical label assigned to every device on a network. It serves two functions: identifying the device (who) and providing its location on the network (where) — like a combination of your name and home address.

Subnet Mask and CIDR Notation
📖 What is Subnetting?

A subnet mask divides an IP address into the network portion (which subnet you're on) and the host portion (which specific device). CIDR notation (e.g., /24) is a shorthand that tells you how many bits are used for the network portion.

  CIDR NOTATION EXPLAINED

  192.168.1.0/24
                ──
                24 bits = Network part
                8 bits  = Host part

  Subnet mask: 255.255.255.0
  Hosts: 2^8 - 2 = 254 usable hosts
  Range: 192.168.1.1 → 192.168.1.254
  Broadcast: 192.168.1.255

  ─────────────────────────────────
  CIDR CHEAT SHEET:
  /8   → 255.0.0.0       → ~16.7M hosts
  /16  → 255.255.0.0     → 65,534 hosts
  /24  → 255.255.255.0   → 254 hosts   ← most common
  /25  → 255.255.255.128 → 126 hosts
  /30  → 255.255.255.252 → 2 hosts  (point-to-point)
  /32  → 255.255.255.255 → 1 host   (single device)

  Security Use Case:
  Segment VLAN_HR = 10.0.1.0/24
  Segment VLAN_IT = 10.0.2.0/24
  Firewall blocks traffic between VLANs
  → Malware in HR can't spread to IT
Static vs Dynamic IP
🔒

Static IP

Manually configured. Never changes. Used for servers, printers, cameras. Easier to firewall-rule — attackers know exactly where to find it.

🔄

Dynamic IP (DHCP)

Automatically assigned by DHCP server. Changes periodically. Used for workstations and phones. DHCP starvation is a real attack.

04 — PORTS & PROTOCOLS
Ports are the doors; protocols are the languages. Master both and you understand how every application communicates over networks.
📖 What is a Port?

A port is a virtual communication endpoint, numbered 0–65535. If an IP address is the building address, the port is the apartment number. One server (one IP) can run dozens of services simultaneously because each uses a different port number.

Key Protocols Explained
🌐

HTTP/HTTPS

HyperText Transfer Protocol. HTTP=unencrypted (port 80), HTTPS=encrypted with TLS (port 443). All web browsing.

🔑

SSH

Secure Shell. Encrypted remote terminal access. Replaced Telnet. Uses port 22. Supports key-based auth.

📧

SMTP/IMAP/POP3

Email protocols. SMTP sends (port 25/587). IMAP retrieves without deleting (143/993). POP3 downloads & deletes (110/995).

🏠

DHCP

Assigns IP addresses automatically. DORA process: Discover, Offer, Request, Acknowledge. Ports 67/68.

📁

FTP/SFTP

File transfer. FTP (21) is plaintext — avoid. SFTP (22) is SSH-based and encrypted — use this.

📊

SNMP

Simple Network Management Protocol. Monitors network devices. Versions 1/2c use community strings (weak). Use SNMPv3.

05 — THE OSI MODEL
The 7-layer reference model that every security professional lives by. Every attack, defense tool, and protocol maps to a specific layer.
📖 Definition

The OSI (Open Systems Interconnection) Model is a conceptual framework developed by ISO that divides network communication into 7 distinct layers. Each layer has a specific job, talks only to the layers directly above/below it, and uses specific protocols. Knowing the OSI model means knowing exactly where an attack happens and what tools defend against it.

💡 Analogy — Sending a Package Internationally

Layer 7: You write the letter (your app creates data). Layer 6: Translate language (encoding). Layer 5: Open conversation with recipient. Layer 4: Choose delivery service (reliable FedEx = TCP, fast courier = UDP). Layer 3: Write the address (IP). Layer 2: Local post office handles it (MAC). Layer 1: The truck physically drives it (cables/Wi-Fi).

06 — TCP/IP MODEL
The real-world networking stack that actually runs the internet — simpler than OSI but what's actually implemented in every device.
📖 Definition

The TCP/IP Model (also called the Internet Model or DoD Model) is the practical 4-layer framework that governs how data is transmitted across the Internet. While OSI is a theoretical reference model, TCP/IP is what's actually implemented in operating systems and network hardware today.

  2. Data Link       ─┐
  1. Physical        ─┴─► 1. Network Access      Ethernet, Wi-Fi, PPP

  OSI = theoretical reference model (standardization)
  TCP/IP = what actually runs on every computer and router

Packet Structure and Encapsulation
  PACKET ENCAPSULATION — Layer by Layer

  Application Layer creates: [DATA]

  Transport Layer adds TCP header:
  [TCP Header | Source Port | Dest Port | Seq# | Flags][DATA]
  This unit is called a SEGMENT

  Internet Layer adds IP header:
  [IP Header | Src IP | Dst IP | TTL | Protocol][TCP Hdr][DATA]

  This unit is called a FRAME

  0110100010110001001... (raw bits on wire/wireless)

When you capture network traffic with Wireshark, you're seeing these exact headers. Defenders analyze headers to detect attacks. Attackers forge headers to spoof source IPs. Understanding packet structure is foundational for both roles.

07 — TCP vs UDP DEEP DIVE
Two transport protocols with fundamentally different philosophies. Knowing when and why each is used is critical for understanding attacks and defenses.
  TCP (Transmission Control Protocol)   UDP (User Datagram Protocol)

  Connection-ORIENTED                    Connection-LESS
  Must handshake before sending          Just sends — no setup
  Reliable — confirms every packet       No confirmation
  In-order delivery guaranteed           Packets may arrive out of order
  Flow control and congestion control    No flow control
  Slower (overhead)                      Faster (minimal overhead)
  Used by: HTTP, SSH, FTP, SMTP         Used by: DNS, VoIP, Gaming, Video

  Analogy:
  TCP = Certified mail with tracking    UDP = Dropping a postcard in a mailbox
        "Did it arrive? Confirm."             "Hope it gets there."
The TCP Three-Way Handshake
SYN

Client → Server: SYN (Synchronize)

Client sends a packet with SYN flag set and a random Initial Sequence Number (ISN). "I'd like to connect. My sequence starts at 1000."

S/A

Server → Client: SYN-ACK (Synchronize-Acknowledge)

Server acknowledges (ACK=1001) and sends its own SYN with its ISN. "Got it. I'm ready. My sequence starts at 4000."

ACK

Client → Server: ACK (Acknowledge)

Client acknowledges server's SYN (ACK=4001). Connection is now fully established. Data can flow in both directions.

  TCP HANDSHAKE TIMELINE

  CLIENT                                    SERVER
    │                                          │
    │──── SYN (seq=1000) ─────────────────────►│
    │                                          │
    │◄─── SYN-ACK (seq=4000, ack=1001) ───────│
    │                                          │
    │──── ACK (ack=4001) ─────────────────────►│

    │══════════ DATA TRANSFER ════════════════│
    │                                          │

    │◄─── FIN-ACK ────────────────────────────│  (4-way closing handshake)
    │──── ACK ────────────────────────────────►│

  SYN FLOOD ATTACK:
  Attacker → Server: SYN (spoofed IP) [×100,000]
  Server: Allocates memory for half-open connections, waits...
  Server: Runs out of memory → CRASHES (Denial of Service)
  Defense: SYN cookies, rate limiting, Firewall rules

07b — TCP THREE-WAY HANDSHAKE
The most important handshake in networking. Every TCP connection starts here — and some of the most common attacks exploit exactly this process.
📖 Definition

The TCP Three-Way Handshake is the connection setup process used by TCP (Transmission Control Protocol) before any data can be exchanged between two devices. It ensures both sides are ready, synchronized, and agree on sequence numbers. Without completing this handshake, no TCP connection exists.

💡 Real World Analogy — Phone Call

Imagine calling a friend:
You: "Hello? Can you hear me?" (SYN — I want to connect)
Friend: "Yes I hear you! Can you hear me?" (SYN-ACK — I acknowledge you, and I'm ready too)
You: "Yes! Perfect." (ACK — confirmed, let's talk)
Now the conversation (data transfer) begins. That's exactly the TCP handshake.

Step-by-Step Breakdown
1

SYN — "I want to connect"

The client sends a TCP segment with the SYN (synchronize) flag set. It also picks a random Initial Sequence Number (ISN) — let's say 1000. This number is used to track packets in order.

Client → Server: [SYN] seq=1000

2

SYN-ACK — "Ready! Here's my info too"

The server replies with both SYN and ACK flags set. It acknowledges the client's sequence number (ACK = 1001, meaning "I got up to 1000, send me 1001 next") and picks its own random ISN — say 4000.

Server → Client: [SYN-ACK] seq=4000, ack=1001

3

ACK — "Got it. Let's go!"

The client acknowledges the server's sequence number (ACK = 4001). No data yet — just the final confirmation. The connection is now fully established (ESTABLISHED state) and data transfer can begin.

Client → Server: [ACK] ack=4001

How Attackers Exploit the Handshake — SYN Flood Attack
⚠️ SYN Flood — DoS Attack

The SYN flood attack exploits the fact that after receiving a SYN, the server allocates memory for a half-open connection and waits for the final ACK. If the ACK never comes, the server's connection table fills up and it can't accept any new legitimate connections — effectively a Denial of Service (DoS).

  NORMAL vs SYN FLOOD ATTACK

  NORMAL (3 packets, connection established):
  Client → Server: SYN
  Server → Client: SYN-ACK  ← Server allocates memory, waits for ACK
  Client → Server: ACK       ← ACK arrives, connection complete ✓

  SYN FLOOD ATTACK:
  Attacker → Server: SYN (src IP = 1.2.3.4 — SPOOFED)
  Server → 1.2.3.4: SYN-ACK ← Memory allocated, waiting...
  ACK never arrives (1.2.3.4 doesn't exist)

  Attacker → Server: SYN (src IP = 5.6.7.8 — SPOOFED)
  Attacker → Server: SYN (src IP = 9.10.11.12 — SPOOFED)
  Attacker → Server: SYN × 100,000 / second

  Server's half-open connection table:
  ┌──────────────────────────────────────┐
  │ 1.2.3.4   : waiting for ACK...       │
  │ 5.6.7.8   : waiting for ACK...       │
  │ 9.10.11.12: waiting for ACK...       │
  │ [TABLE FULL — can't accept new conns]│
  └──────────────────────────────────────┘
  Result: REAL users can't connect → Service DOWN

  DEFENSES:
  ✓ SYN Cookies — server doesn't allocate memory until ACK received
  ✓ Rate limiting — firewall drops excessive SYNs from one source
  ✓ Firewall rules — block known scanning IPs
  ✓ Load balancers — distribute and absorb attack traffic

💡 Why Sequence Numbers Matter for Security

In the early internet, TCP sequence numbers were predictable. Attackers could guess the next sequence number and inject forged packets into a TCP session — called a TCP Session Hijacking attack. Modern OS implementations use cryptographically random ISNs to prevent this.

08 — DNS DEEP DIVE
The internet's phone book — and one of the most attacked protocols in existence. Understand DNS to understand a huge attack surface.
📖 Definition

DNS (Domain Name System) translates human-readable domain names (google.com) into machine-readable IP addresses (142.250.80.46). Without DNS, you'd need to memorize IP addresses for every website. DNS operates over port 53, primarily using UDP for queries.

DNS Resolution — Step by Step
1

Browser checks local cache

Before any network request, the browser checks if it already knows the IP (from a recent visit). Operating system also has a DNS cache (ipconfig /displaydns on Windows).

2

Ask the Recursive Resolver

If not cached, your device asks the configured DNS server (e.g., 8.8.8.8 Google, 1.1.1.1 Cloudflare, or your ISP's). The resolver does the heavy lifting.

3

Resolver asks Root Nameserver

13 sets of root nameservers globally. Resolver asks: "Who handles .com?" Root says: "Ask the .com TLD nameserver."

4

TLD Nameserver → Authoritative NS

.com TLD server says: "google.com's authoritative nameserver is ns1.google.com." Resolver then asks Google's own nameserver.

5

Authoritative Nameserver responds

Google's ns1.google.com answers: "142.250.80.46" — the real IP. Resolver caches it, sends it back. Browser connects. Entire process: ~20–100ms.

  DNS RESOLUTION FLOW

  [Your Browser] ──1──► [OS DNS Cache]
        │ (cache miss)
        ▼
  [Recursive Resolver] ← 8.8.8.8 or 1.1.1.1
        │ (cache miss)
        ├──2──► [Root Nameserver] ──► ".com TLD: ask ns.verisign.com"
        ├──3──► [.com TLD Server]  ──► "google.com: ask ns1.google.com"
        └──4──► [Google Auth NS]   ──► "google.com = 142.250.80.46 ✓"
        │
        └──5──► [Browser] ──► Connect to 142.250.80.46:443

  Recursive = resolver does all the work for you
  Iterative = you do each step yourself (rare, for advanced queries)

📖 What is NAT?

NAT (Network Address Translation) allows multiple devices with private IP addresses to share a single public IP address when accessing the internet. Your router performs NAT — translating your device's 192.168.1.x address to your ISP-assigned public IP before packets leave your network.

  HOW NAT WORKS

  Private Network              Router/NAT          Internet
  ─────────────────            ─────────────       ────────────

  PC1: 192.168.1.10   ──┐
  PC2: 192.168.1.11   ──┼──► NAT Translation ──► Public IP: 203.0.113.50
  PC3: 192.168.1.12   ──┘     Table:              → Google
  Phone: 192.168.1.20 ─┘
                              ┌────────────────────────────────┐
                              │ Private IP:Port → Public IP:Port│
                              │ 192.168.1.10:5000 → 203.0.113.50:41234│
                              │ 192.168.1.11:3000 → 203.0.113.50:41235│
                              └────────────────────────────────┘
  
  Google sees: ONE request from 203.0.113.50
  Your ISP sees: One public IP for your whole house
  Security benefit: Internal IPs hidden from internet (obscurity)
  Security risk: Once attacker is inside, NAT provides NO protection
Routing Basics
📖 What is Routing?

A router examines every packet's destination IP and consults its routing table to determine the best next hop. This process, repeated at every router along the path, guides packets from source to destination across the internet.

  ROUTING TABLE — Router's Decision Engine

  $ route -n    (Linux routing table example)

  Destination    Gateway        Genmask         Interface
  0.0.0.0        192.168.1.1    0.0.0.0         eth0  ← Default gateway (internet)
  10.0.0.0       0.0.0.0        255.0.0.0       eth1  ← Direct: 10.x.x.x network
  192.168.1.0    0.0.0.0        255.255.255.0   eth0  ← Direct: local network
  192.168.2.0    10.0.0.1       255.255.255.0   eth1  ← Route to remote subnet

  Decision logic:
  1. Check routing table for matching destination
  2. Most specific match wins (/32 beats /24 beats /0)
  3. Send packet to next-hop gateway
  4. Repeat at each router until destination reached

  Default Gateway = router that handles all unknown destinations
  0.0.0.0/0 = "everything else" → send to ISP

TypeDescriptionCommon Use
Static NATOne-to-one mapping: one private IP = one public IPServers that need consistent public IP (web, mail)

⚠️ Security Note — Port Forwarding Risk

Port forwarding punches a hole through NAT — exposing an internal service directly to the internet. Every exposed port is a potential attack surface. Use VPN instead of port forwarding for remote access when possible. If you must port forward, use strong authentication and keep the service patched.

14 — FIREWALLS & NETWORK SECURITY
The gatekeepers of network security — how firewalls work, the different types, and why they're not enough alone.
📖 Definition

A firewall is a network security device (hardware or software) that monitors incoming and outgoing network traffic and decides whether to allow or block based on a defined set of security rules. It creates a barrier between trusted internal networks and untrusted external networks.

Firewall Types — Evolution
  FIREWALL EVOLUTION

  1. Packet Filtering Firewall (Layer 3/4)
  Rules: IF src_ip=X AND dst_port=80 → ALLOW
  Checks: IP, port, protocol only
  Weakness: Can't see context — stateless
  Example: iptables basic rules

  2. Stateful Inspection Firewall (Layer 4)
  Tracks connection states (NEW, ESTABLISHED, RELATED)
  Knows if packet is part of legitimate session
  Much more secure than packet filtering
  Example: Most modern firewalls include this

  3. Application-Layer (Proxy) Firewall (Layer 7)
  Understands application protocols (HTTP, FTP, DNS)
  Can inspect actual content, not just headers
  Slowest but most intelligent
  Example: Squid proxy, Blue Coat

  4. Next-Generation Firewall (NGFW)
  All of the above PLUS:
  - Deep Packet Inspection (DPI)
  - Application awareness (identify app, not just port)
  - IDS/IPS built in
  - SSL inspection (decrypt HTTPS to inspect)
  - User identity awareness
  Example: Palo Alto, Fortinet, CheckPoint

  5. WAF — Web Application Firewall
  Specifically protects web apps at Layer 7
  Blocks SQLi, XSS, CSRF, path traversal
  Example: Cloudflare WAF, AWS WAF, ModSecurity
Firewall Rule Logic
IPTABLES — LINUX FIREWALL RULES
# View current rules iptables -L -n -v # Allow established/related connections iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT # Allow SSH from specific IP only iptables -A INPUT -p tcp --dport 22 -s 192.168.1.50 -j ACCEPT # Allow HTTPS from anywhere iptables -A INPUT -p tcp --dport 443 -j ACCEPT # Drop Telnet (never should be open) iptables -A INPUT -p tcp --dport 23 -j DROP # Block specific attacker IP iptables -A INPUT -s 185.220.101.45 -j DROP # Default deny everything else (golden rule) iptables -P INPUT DROP iptables -P FORWARD DROP

15 — NETWORK MONITORING & LOGGING
You can't defend what you can't see. Network monitoring and logging are how security teams detect attacks in real time.
📖 Definition

Network monitoring is the continuous observation of a network for performance, availability, and security events. Network logging is recording network events (connections, alerts, traffic) for later analysis. Together, they form the visibility layer that SOC analysts rely on.

What to Monitor
🔥

Firewall Logs

Every allowed and denied connection. Shows attack attempts, blocked threats, and policy violations. First place analysts look.

🚨

IDS/IPS Alerts

Signature matches against known attack patterns. Snort/Suricata rules fire when attack traffic detected. Critical alerts need immediate triage.

🌐

DNS Logs

Every domain lookup. Reveals C2 communication, malicious domain access, and DNS tunneling. Often overlooked but valuable.

🔀

NetFlow / IPFIX

Summary of all network flows (source, destination, bytes, duration). Detects anomalies without full packet content.

📡

DHCP Logs

Which MAC got which IP at what time. Critical for identifying rogue devices and attributing traffic to specific machines.

🕵️

Proxy / Web Logs

All HTTP/HTTPS requests from internal users. Reveals malware downloads, C2 beaconing, and data exfiltration.

16 — NETWORK ATTACKS OVERVIEW
How attackers abuse networking — port scanning, man-in-the-middle, ARP spoofing, DNS poisoning, and DDoS. Know the attack to build the defense.
⚠️ Ethical Disclaimer

This module describes attacks for educational purposes only. Understanding how attacks work is essential for defenders. All techniques described here should only be practiced in authorized lab environments (HackTheBox, TryHackMe, your own VMs). Using these techniques against systems you don't own is illegal.

1. Port Scanning
📖 What It Is

Port scanning sends packets to target ports to determine which are open (running services), closed (no service), or filtered (firewall). It's the first step in almost every network attack — attackers map what's available before deciding what to exploit.

NMAP — PORT SCANNING
# Basic TCP scan (top 1000 ports) nmap 192.168.1.1 # Detect service versions and OS nmap -sV -O 192.168.1.1 # Full scan all 65535 ports nmap -p- 192.168.1.1 # Stealth SYN scan (doesn't complete handshake) nmap -sS 192.168.1.1 # Scan entire subnet nmap -sn 192.168.1.0/24 # Ping sweep # Example output: PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 8.2 80/tcp open http nginx 1.18 443/tcp open https 3306/tcp open mysql MySQL 8.0 ← Should NOT be exposed!
🔐 Defense

Host-based firewalls block scan attempts. IDS signatures detect Nmap scan patterns. Honeypots alert when scanned. Monitor logs for sequential port connection attempts from single source.

2. Man-in-the-Middle (MITM) Attack
  MAN-IN-THE-MIDDLE ATTACK

  NORMAL:   Alice ──────────────────────────► Bob

  ATTACKED: Alice ──► ATTACKER ──────────────► Bob
                      (intercepts,                (receives
                       reads/modifies)             modified data)

  Common MITM Techniques:
  1. ARP Spoofing     — Poison ARP cache, intercept LAN traffic
  2. Evil Twin AP     — Fake Wi-Fi hotspot that proxies your traffic
  3. SSL Stripping    — Downgrade HTTPS to HTTP (use HSTS to prevent)
  4. BGP Hijacking    — Reroute internet traffic at ISP level
  5. DNS Spoofing     — Redirect domains to attacker-controlled server

  What attacker can see/do:
  - Read all unencrypted HTTP traffic (passwords, cookies)
  - Inject malicious content into web pages
  - Steal session cookies to impersonate victim
  - Intercept credentials even on "secure" networks
3. ARP Spoofing / Poisoning
  ARP SPOOFING EXPLAINED

  Normal ARP:
  PC asks: "Who has IP 192.168.1.1? Tell me your MAC"
  Router: "192.168.1.1 is at MAC AA:BB:CC:DD:EE:FF"
  PC stores: 192.168.1.1 → AA:BB:CC:DD:EE:FF in ARP cache

  ARP Spoofing Attack:
  Attacker sends FAKE ARP replies (without being asked):
  → To PC:     "192.168.1.1 (Router) is at MY MAC 11:22:33:44:55:66"
  → To Router: "192.168.1.100 (PC) is at MY MAC 11:22:33:44:55:66"

  Result:
  PC → [Attacker] → Router    (attacker intercepts all PC traffic)
  Router → [Attacker] → PC    (full MITM established)

  Detection: Watch for duplicate ARP replies, IP-MAC conflicts
  Prevention: Dynamic ARP Inspection (DAI) on managed switches
                Static ARP entries for critical devices
4. DNS Poisoning (Cache Poisoning)
  DNS CACHE POISONING

  Normal flow:
  User → Resolver → Google NS → "google.com = 142.250.80.46" ✓

  Attack flow:
  User → Resolver → [Resolver asks root NS]
  Attacker floods resolver with FAKE responses:
  "google.com = 10.0.0.99 (attacker's server)"
  If attacker's fake response arrives before real one:
  Resolver caches the WRONG IP
  ALL users using this resolver → redirected to attacker's site

  Impact:
  - Phishing: Fake login page steals credentials
  - Malware distribution: Fake site serves malware
  - Full invisibility: User sees correct URL in browser

  Prevention: DNSSEC (cryptographically signs DNS records)
               Use trusted DNS (1.1.1.1, 8.8.8.8 with DoH/DoT)
5. DDoS — Distributed Denial of Service
  DDOS ATTACK ARCHITECTURE

  Attacker
      │ (Commands via C2 server)
      │
  ┌───▼──────────────────────────────────────────┐
  │        BOTNET (thousands of compromised PCs)  │
  │  Bot1  Bot2  Bot3  Bot4  Bot5  Bot6  Bot7... │
  └───┬──────────────────────────────────────────┘
      │ (Millions of requests per second)
      ▼
  TARGET SERVER — overwhelmed — crashes — service down

  Common DDoS Types:
  Volumetric    — Saturate bandwidth (UDP flood, ICMP flood)
  Protocol      — Exhaust server resources (SYN flood)
  Application   — Target Layer 7 (HTTP GET flood, Slowloris)

  Notable attacks:
  GitHub 2018: 1.35 Tbps (memcached amplification)
  AWS 2020:    2.3 Tbps (largest recorded)

  Defenses:
  - Cloudflare / Akamai DDoS protection
  - Rate limiting and SYN cookies
  - BGP blackholing
  - Anycast routing to distribute traffic
